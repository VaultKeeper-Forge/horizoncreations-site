# Shopify Backend Architecture

This project keeps Horizon Creations as the primary branded site.

Shopify is being prepared as a future backend system of record for items that are actually for sale, but no Shopify data is rendered on the frontend yet.

## Current architecture

- Frontend site: static pages generated by `scripts/build-site.mjs`
- Gallery content: `content/standard-pieces/`, `content/custom-pieces/`, `content/workbench/`
- Product staging: `content/product-staging/`
- Shopify backend utilities: `scripts/shopify/`

## Why product staging is separate

The gallery content model is for visual/site presentation.

The staging model is for commerce prep and approval workflows.

That separation makes it possible to:

- keep gallery-only content off Shopify
- keep custom one-off work out of a storefront unless explicitly reclassified
- review sellable items before export
- store Shopify sync state without polluting the frontend content model

## Environment variables

Credentials are environment-variable only.

Use shell environment variables or a local ignored `.env.local` file.

Supported vars:

- `SHOPIFY_STORE_DOMAIN`
- `SHOPIFY_ADMIN_ACCESS_TOKEN`
- `SHOPIFY_STOREFRONT_ACCESS_TOKEN`
- `SHOPIFY_API_VERSION`

See `.env.example` for the placeholder format.

## Commands

Validate staged products:

```powershell
npm run shopify:validate
```

Preview product payloads locally:

```powershell
npm run shopify:preview
```

Dry-run export:

```powershell
npm run shopify:export:dry-run
```

Attempt real export:

```powershell
npm run shopify:export
```

## Approval-first workflow

No content exports automatically.

For a staged product to be exportable, it should be:

- `kind: "sellable-product"`
- `sellable: true`
- `workflow.readyForReview: true`
- `workflow.approvedForExport: true`

The sample sellable entry is intentionally left unapproved so nothing can publish by accident.

## Export behavior

The exporter is defensive:

- it validates staged entries first
- it skips anything non-sellable
- it skips anything not approved
- it fails clearly if you try a real export without Shopify credentials
- it never affects the site build

## Shopify API path

The scaffold is built around the Shopify Admin GraphQL API.

Relevant official docs used for this preparation:

- [GraphQL Admin API endpoint and auth](https://shopify.dev/docs/api/admin-graphql/latest)
- [productCreate mutation](https://shopify.dev/docs/api/admin-graphql/latest/mutations/productcreate)
- [productSet mutation](https://shopify.dev/docs/api/admin-graphql/latest/mutations/productSet)

Inference from those docs:

- `productCreate` is the right future path for creating draft/unpublished products from staged local content
- `productSet` is the likely future path for safe update/sync operations once products already exist in Shopify

## What remains before real Shopify activation

- create a Shopify store / custom app
- add real credentials to environment variables
- confirm required API scopes such as `write_products`
- decide how product media should upload:
  - remote image URLs
  - or a later local-to-remote upload step
- decide whether future live site sections should read from Shopify Storefront API, local build data, or a hybrid source
- optionally add a real sync/update flow for already-exported products

## Practical future use

Later, the intended flow is:

1. Add new product photos/content locally
2. Have Codex help categorize and write metadata
3. Stage the item under `content/product-staging/`
4. Run validation and payload preview
5. Mark the item approved
6. Run export to create a Shopify draft product
7. Review in Shopify before any customer-facing publish step
